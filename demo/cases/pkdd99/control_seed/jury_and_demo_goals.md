# Jury notes y objetivos de demo — PKDD’99

## Qué debe quedar claro al público
- el tiempo cambia radicalmente el problema;
- hay que fijar fecha de corte y unidad de predicción antes de crear features;
- la demo no busca “exprimir el dataset”, sino plantear correctamente el proyecto;
- los repos de control y ejecución se coordinan con un flujo simple.

## Qué debería poder contestar el equipo durante la demo
- cuál es la unidad de predicción;
- qué datos quedan prohibidos por ser posteriores a la fecha de préstamo;
- cómo se resumen las transacciones históricas;
- cómo cambia la validez del modelo si se rompe la restricción temporal;
- qué implicaciones tiene esto para industrialización.

## Señales de una buena ejecución
- documentación temporal clara;
- riesgos de leakage explícitos;
- backlog ordenado;
- pasos pequeños y verificables.
